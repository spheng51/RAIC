# Execution Plan: v0.4.0 Reliable Adaptive Learning Platform

Date: 2026-05-17
Target release: `v0.4.0`
Milestone name: `v0.4.0 Reliable Adaptive Learning Platform`

## Summary

`v0.4.0` builds on the tagged `v0.3.0` public release. The release goal is to make adaptive learning more reliable and governable before widening student-facing behavior.

Status: implementation complete on `main`; release closeout is limited to versioning, release notes, final QA evidence, and tagging.

The release sequence stays conservative:

1. Land a documentation and release-hygiene patch.
2. Complete Provider Composer coverage for scene-generation routes.
3. Harden strict scenario-managed provider routes so they fail closed.
4. Add private teacher/internal learning analytics from existing session context and reflection data.
5. Start adaptive student behavior only as an explicitly reviewed, feature-flagged beta.

## Current Baseline

- `v0.3.0` is tagged and published from `main`.
- `v0.4.0` implementation slices have landed on `main`: release hygiene, Provider Composer scene routing, fail-closed provider hardening, teacher/internal analytics, and closeout stabilization.
- Production is live on `open-raic.com`.
- Vercel runtime policy is Node `24.x`; local release commands should run under Node 24.
- Existing adaptive classroom intelligence is teacher-only and fail-open for first-run, anonymous, public demo, student, and classroom-cookie flows.
- Provider scenario routing now covers verification, image, video, TTS, transcription, web-search, and scene-generation paths.

## Milestone A: Provider Composer Completion

Status: complete.

Scope:

- Add scenario-managed routing to:
  - `/api/generate/scene-outlines-stream`
  - `/api/generate/scene-content`
  - `/api/generate/scene-actions`
- Preserve all public request and response payload shapes.
- Emit provider-scenario telemetry with:
  - `scenarioProfileId`
  - `taskBucket`
  - `routeId`
  - `selectedProviderId`
  - `selectedModelId`
  - `fallbackProviderId`
  - `fallbackModelId`
  - `fallbackReason`
  - `validationStatus`

Acceptance:

- Scene outline, content, and action generation use the same validation and telemetry contract as existing managed media/provider routes.
- Client-requested provider hints cannot bypass governed scenario policy.
- Focused route tests prove payload compatibility and scenario selection.

## Milestone B: Fail-Closed Provider Hardening

Status: complete.

Scope:

- Remove silent unmanaged fallback where a route is explicitly scenario-managed.
- Keep best-effort fallback only where the route contract explicitly allows it.
- Normalize denial telemetry and governed provider errors.

Acceptance:

- Invalid managed candidates return governed 4xx responses.
- Tests prove unmanaged fallback resolvers are not called after strict scenario failure.
- Existing `/api/server-providers` and public provider payloads remain unchanged.

## Milestone C: Learning Analytics + Reflection

Status: complete for private teacher/internal aggregate analytics.

Scope:

- Add teacher/internal aggregate learning quality signals from existing session context and reflection records.
- Keep anonymous, public, and student analytics off by default.
- Use deterministic fixtures for reporting tests.
- Apply platform retention policy to any reflection-derived analytics artifacts.

Acceptance:

- Internal analytics payloads avoid learner-identifying data unless explicitly reviewed and approved.
- Retention tests cover analytics artifacts.
- No public/student API shape changes.

## Milestone D: Adaptive Student Beta

Status: deferred to a later reviewed milestone.

Scope:

- Begin only after privacy, consent, retention, and public API review are written down.
- Gate student-facing adaptation behind feature flags.
- Add rollback controls and clear fallback behavior.

Acceptance:

- Student adaptation is opt-in or policy-approved.
- Tests prove teacher/private analytics do not leak into student flows.
- Disabling the flag restores `v0.3.0` student behavior.

## Release Gate

Every implementation milestone must pass focused slice tests first. Before tagging `v0.4.0`, run the full clean-main gate under Node 24:

- `corepack pnpm run secrets:scan`
- `corepack pnpm run ops:drift`
- `corepack pnpm run check`
- `corepack pnpm lint`
- `corepack pnpm run build`
- `corepack pnpm run test:mirofish:gate`
- `corepack pnpm run test:mirofish:e2e`
- `CI=1 corepack pnpm run test:e2e`
- `corepack pnpm run benchmark:milestone`
- `corepack pnpm run ops:verify`

After production deploy:

- `corepack pnpm run smoke:production:milestone`
- `corepack pnpm run smoke:production:classroom`

## Rollback Conditions

- Provider Composer: rollback if scenario selection causes provider availability drift, fallback loops, unmanaged policy bypass, or measurable route latency regression.
- Fail-closed hardening: rollback if governed provider errors become ambiguous or block routes outside the managed scenario scope.
- Analytics: rollback if payloads contain unapproved learner-identifying data or retention cleanup misses analytics artifacts.
- Student beta: rollback if private teacher/internal signals are exposed to student flows or feature flags cannot restore baseline behavior.

## Out Of Scope

- Discord scheduled-class sync.
- Public/student API changes outside the explicitly reviewed student beta.
- Promoting archived dirty worktree changes without a focused PR.
